# EP-01 — User Authentication & Access

## Goal
Enable anyone to register and access the platform with the appropriate role (Buyer, Seller, or Admin) for the duration of a specific event.

## Public Access (no login required)
- Direct Donations (EP-04) — donate, view progress bar, view leaderboard
- Event Timeline & Countdown (EP-08) — countdown, key dates

## Login Required
- Browse shops and products (EP-07)
- Place orders (EP-07)
- Admin and Seller features (EP-02, EP-03, EP-05)

## In Scope
- Self-registration via email + 6-digit numeric passcode — instant activation, no email verification (Phase 1)
- SSO via Azure Active Directory / Entra ID (Phase 2 — OIDC, no code changes required)
- Role-based access control: Buyer (default on registration), Seller (event-scoped, Admin-assigned), Admin (seeded in code or promoted by another Admin)
- Event-scoped Seller access — read-only after event ends; re-granted per new event
- Session management, logout (invalidates all active sessions across tabs)
- Dormant state — when no event is active, authenticated users see the previous event's donation results
- In-app notification to Sellers when their edit access is revoked at event end (low priority)

## Out of Scope
- Email verification at registration (Phase 1)
- Passcode reset (Phase 2 SSO handles this via AAD)

## Auth Design Notes
- Use OIDC/JWT with standard claims only (sub, email, iat, exp, aud, iss) — required for Phase 2 AAD SSO swap
- Roles stored in the application DB (not in the token) — avoids vendor lock-in and supports event-scoped role logic
- Phase 2 migration: swap Supabase Auth OIDC provider config to AAD; no application logic changes
- First Admin is seeded in code at deployment; system must always retain at least one Admin (cannot delete the last Admin)
- Unlimited Admin accounts allowed

## Validation Rules
- **Email:** Valid format required; must be unique — if already registered, show "Email already in use"
- **Passcode:** Exactly 6 digits, numeric only; re-enter field must match on sign-up
- **Deactivated account login:** Show "Your account has been deactivated. Please contact the event organiser." Block access.
- **Role re-evaluation:** Per-request — role is resolved from the app DB on every authenticated request, so access changes (e.g. event close → read-only) take effect immediately without requiring re-login

## Roles Involved
- Visitor (unauthenticated) — donations + timeline only
- User / Buyer (logged in) — full browse + order + donate
- Seller (event-scoped, Admin-assigned)
- Admin (pre-seeded)

## User Stories

| # | Priority | Story |
|---|---|---|
| US-01.01 | High | As a **visitor**, I want to sign up with my name, email and a 6-digit passcode, so that I can create an account and access the platform |
| US-01.02 | High | As a **user**, I want to log in with my email and passcode, so that I can browse, place orders and donate |
| US-01.03 | High | As a **user**, I want to log out, so that my account stays secure |
| US-01.04 | Medium | As an **Admin**, I want seller edit access to automatically become read-only when an event ends, so that access is managed without manual cleanup |
| US-01.05 | Medium | As an **Admin**, I want to promote a registered user to Admin, so that I can delegate platform management |
