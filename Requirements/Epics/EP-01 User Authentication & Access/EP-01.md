# EP-01 — User Authentication & Access

## Goal
Enable employees to securely log in and access the platform with the appropriate role (Buyer, Seller, or Admin) for the duration of a specific event.

## In Scope
- Employee login via email/password (Phase 1: Supabase Auth)
- SSO via Azure Active Directory / Entra ID (Phase 2 — OIDC, no code changes required)
- Role-based access control: Buyer, Seller, Admin
- Event-scoped access — a user's Seller role is active for one event only; access is revoked when the event ends
- Session management and logout
- Admin-provisioned accounts only — no self-registration

## Out of Scope
- Self-registration by employees
- Password reset (Phase 2 SSO handles this via AAD)
- Persistent seller accounts across multiple events

## Auth Design Notes
- Use OIDC/JWT with standard claims from day one — required for Phase 2 AAD SSO swap
- Roles stored in the application DB (not in the token) — avoids vendor lock-in and supports event-scoped role logic
- Phase 2 migration: swap Supabase Auth OIDC provider config to AAD; no application logic changes

## Roles Involved
- Employee (Buyer)
- Employee (Seller)
- Admin
