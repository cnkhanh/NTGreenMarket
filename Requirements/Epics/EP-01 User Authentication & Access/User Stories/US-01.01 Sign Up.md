# US-01.01 Sign Up

### User Story ID
`US-01.01`

### User Story Name
Sign Up

### User Story
As a **visitor**, I want to sign up with my name, email, location, and a 6-digit passcode, so that I can create an account and access the platform.

---

### Pre-conditions
| Pre-condition ID | Description |
|------------------|-------------|
| PR01 | The visitor is unauthenticated and has not previously registered with the provided email address. |
| PR02 | The sign-up page is publicly accessible without login. |

---

### Workflow/Activity Diagram

1. Visitor navigates to the Sign Up page.
2. System attempts IP geolocation and pre-fills the Location dropdown with the closest matching office city.
3. Visitor enters their name, email address, confirms or changes their location, enters a 6-digit numeric passcode, and re-enters the passcode to confirm.
4. System validates all fields: name not empty; email format valid and unique; location selected; passcode exactly 6 digits; re-entered passcode matches.
5. **If validation fails:** System displays inline error message(s) next to the failing field(s). Visitor remains on the Sign Up page.
6. **If validation passes:** System creates the account with the Buyer role, stores the selected location, and activates it immediately.
7. System redirects the visitor to the authenticated home page (Buyer view).

---

### Suggested UI Specifications

| Field Name | Type | Mandatory | Default Value | Description |
|---|---|---|---|---|
| Full name | Text input | Yes | N/A | The user's display name. Max 100 characters. Error: "Please enter your name." |
| Email address | Text input | Yes | N/A | Valid email format (RFC 5322). Max 254 characters. Error: "Please enter a valid email address." |
| Location | Dropdown | Yes | Pre-filled via IP geolocation (best-effort) | Options: Ha Noi, Da Nang, Ho Chi Minh. User must confirm or change. Error: "Please select your location." |
| Passcode | Numeric input (masked) | Yes | N/A | Exactly 6 digits, numeric only. Error: "Passcode must be exactly 6 digits." |
| Re-enter passcode | Numeric input (masked) | Yes | N/A | Must match the passcode field. Error: "Passcodes do not match." |
| Sign Up button | Primary button | Yes | N/A | Submits the form. Disabled while submission is in progress. |
| Phase 1 notice | Informational text | No | N/A | "Forgotten your passcode? Please contact the event organiser." |
| Log In link | Text link | No | N/A | Navigates to the Log In page. |

### Business Acceptance Criteria

**AC01** — Successful registration

**Given** a visitor is on the Sign Up page,
**When** they submit a valid name, unique email, a valid 6-digit passcode, and the matching re-entered passcode,
**Then** their account is created immediately with the Buyer role, and they are redirected to the authenticated home page.

**AC02** — Location pre-fill and validation

**Given** a visitor opens the Sign Up page,
**Then** the system attempts to detect the visitor's city via IP geolocation and pre-selects the closest matching location (Ha Noi / Da Nang / Ho Chi Minh). If detection fails, no location is pre-selected.

**Given** a visitor submits the form,
**When** no location is selected,
**Then** the system displays "Please select your location." and does not create an account.

**AC03** — Name validation

**Given** a visitor submits the form,
**When** the name field is empty,
**Then** the system displays "Please enter your name." and does not create an account.

**AC03** — Email validation

**AC03.1** — Invalid email format

**Given** a visitor submits the form,
**When** the email address is not a valid format,
**Then** the system displays "Please enter a valid email address." and does not create an account.

**AC03.2** — Email already in use

**Given** a visitor submits the form,
**When** the email address is already registered,
**Then** the system displays "Email already in use. Please use a different email or log in." and does not create a duplicate account.

**AC04** — Passcode validation

**AC04.1** — Invalid passcode format

**Given** a visitor submits the form,
**When** the passcode is not exactly 6 numeric digits,
**Then** the system displays "Passcode must be exactly 6 digits." and does not create an account.

**AC04.2** — Passcode confirmation mismatch

**Given** a visitor submits the form,
**When** the re-entered passcode does not match the passcode,
**Then** the system displays "Passcodes do not match." and does not create an account.

**AC05** — Default role assignment

**Given** a new account is created successfully,
**Then** the account is assigned the Buyer role by default. No Seller or Admin role is granted.

**AC06** — Instant activation — no email verification

**Given** the system creates the account,
**Then** the account is active immediately. No email verification step is required before the user can log in.

**AC07** — Passcode reset notice

**Given** a visitor is on the Sign Up page,
**Then** a static notice is displayed informing the user that passcode reset is not available and they should contact the event organiser if they forget their passcode.

### Out of Scope
| OOS ID | Description |
|--------|-------------|
| OOS01 | Email verification — accounts are activated instantly; no verification email is sent. |
| OOS02 | Passcode reset / forgot passcode — out of scope for Phase 1. |
| OOS03 | Social or SSO sign-up (e.g., Google, Microsoft) — Phase 1 uses email and passcode only. |
| OOS04 | Admin self-registration — Admin accounts are seeded in code and cannot be created via this page. |

---

### Non-functional Requirements
| Requirement | Description |
|---|---|
| Security | Passcodes must be hashed using a strong algorithm (e.g., bcrypt) before storage. Plain-text passcodes must never be stored or logged. |
| Performance | Sign-up form submission should complete within 2 seconds under normal load. |
| Cross-platform | Form must be fully functional on mobile and desktop browsers (Chrome, Safari, Edge — latest two versions). |

---

### Dev Handoff Notes

- Auth provider: Supabase Auth (Phase 1). Use standard `signUp` method with the passcode as the password field (Supabase has no native passcode type — treat the 6-digit passcode as a password string, validated client-side for format before submission).
- **Passcode hashing:** Supabase Auth automatically hashes credentials (bcrypt) before storage. The app must never store or log the raw passcode — only pass it to the Supabase `signUp` call and discard it immediately.
- JWT claims: `sub`, `email`, `iat`, `exp`, `aud`, `iss` only. No Supabase-specific extensions in business logic.
- After Supabase creates the auth record, the application must insert a corresponding user record in the app DB (Prisma) with `role = BUYER`, the `name` field, and the `location` field (`HA_NOI` | `DA_NANG` | `HO_CHI_MINH`) from the form input.
- The `name` and `location` fields are stored in the app DB user record (not in the JWT).
- IP geolocation: call a free API (e.g. `https://ipapi.co/json/`) on page load to get `city`. Map city to location enum — treat as suggestion only, never enforce. Handle API failure gracefully (no pre-fill, no error).
- Re-enter passcode validation is client-side only — do not send both values to the server; send only the passcode once validated.
- Phase 2: Supabase Auth OIDC config will be swapped for Azure AD / Entra ID. No application-level code changes should be required if the above JWT constraint is respected.
