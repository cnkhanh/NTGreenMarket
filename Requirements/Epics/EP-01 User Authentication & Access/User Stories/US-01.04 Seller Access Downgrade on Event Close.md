# US-01.04 Seller Access Downgrade on Event Close

### User Story ID
`US-01.04`

### User Story Name
Seller Access Downgrade on Event Close

### User Story
As an **Admin**, I want seller edit access to automatically become read-only when an event ends, so that access is managed without manual cleanup.

---

### Pre-conditions
| Pre-condition ID | Description |
|------------------|-------------|
| PR01 | At least one event exists in the system with one or more Sellers assigned to it. |
| PR02 | The event is currently in an active state (Seller role status = Active for that event). |
| PR03 | The Admin has either closed the event manually (EP-02 US-02.04) or the event end date has passed and the system has evaluated it. |

---

### Workflow/Activity Diagram

**Trigger A — Admin closes event manually (EP-02 US-02.04):**

1. Admin closes the event via the Admin dashboard.
2. System sets event status to Closed.
3. System updates the role status for all Sellers assigned to that event from `Active` → `ReadOnly` in the app DB.
4. System sends an in-app notification to each affected Seller (low priority — see AC05).
5. On the Seller's next request, role is resolved as ReadOnly. No re-login is required.

**Trigger B — Event end date passes:**

1. System evaluates event end date (scheduled job or on-request evaluation).
2. If the current date is past the event end date and the event is still active, the system closes the event automatically.
3. Steps 3–5 above apply identically.

**No-shop path:**

1. If a Seller has no shop registered for the closing event, the role status update still occurs in the app DB.
2. No in-app notification is sent and no visible UI change occurs for that Seller.

---

### Suggested UI Specifications

**Seller view — post-downgrade:**

| Element | Behaviour |
|---|---|
| Shop management page | Visible in read-only mode. All create, edit, and delete controls (buttons, forms) are hidden or disabled. |
| Product listing page | Visible in read-only mode. Add Product, Edit, and Delete actions are hidden or disabled. |
| Order management | Seller can view past orders but cannot accept new orders. Accept / reject order actions are hidden or disabled. |
| Read-only indicator | Optional: a banner or label (e.g., "This event has ended. Your shop is now in read-only mode.") may be displayed on the Seller's shop page. |

**In-app notification (low priority):**

| Element | Behaviour |
|---|---|
| Notification message | "The event has ended. Your shop access is now read-only. You can still view your shop and order history." |
| Delivery | In-app notification (bell icon or notification centre). Email notification is out of scope for Phase 1. |

---

### Business Acceptance Criteria

**AC01** — Access downgrade triggered by Admin event close

**Given** an Admin closes an active event (EP-02 US-02.04),
**When** the event status changes to Closed,
**Then** the role status for all Sellers assigned to that event is updated from Active to ReadOnly in the app DB.

**AC02** — Access downgrade triggered by event end date

**Given** an event's end date has passed,
**When** the system evaluates the event status,
**Then** the event is automatically closed and all assigned Sellers' role status is updated to ReadOnly, identical to the Admin-triggered close.

**AC03** — Immediate access restriction — no re-login required

**Given** a Seller's role status has been set to ReadOnly,
**When** the Seller makes any subsequent request,
**Then** the system resolves the role as ReadOnly from the app DB and restricts access immediately. The Seller does not need to log out and log back in.

**AC04** — Read-only restrictions enforced

**Given** a Seller's role status is ReadOnly for a closed event,
**Then** the Seller can view their shop, product listings, and order history for that event, **and** the Seller cannot create, edit, or delete products or accept new orders. All mutating actions are hidden or disabled in the UI and rejected server-side.

**AC05** — In-app notification on access revocation (low priority)

**Given** a Seller's role status has been set to ReadOnly and the Seller has a shop registered for the event,
**When** the downgrade occurs,
**Then** an in-app notification is sent to the Seller informing them that their access is now read-only.
*Note: This is a low-priority requirement. If deprioritised, the downgrade must still occur silently — the notification is the only optional part.*

**AC06** — Sellers with no shop — silent downgrade

**Given** a Seller has no shop registered for the closing event,
**When** the event is closed,
**Then** the role status update occurs in the app DB, no notification is sent, and there is no visible UI change for that Seller.

**AC07** — Historical data remains accessible

**Given** an event has been closed,
**When** any authenticated user accesses historical data for that event,
**Then** shop listings, product details, and order history remain visible in read-only form for all roles.

---

### Out of Scope
| OOS ID | Description |
|--------|-------------|
| OOS01 | Email notifications to Sellers on access revocation — in-app notification only for Phase 1. |
| OOS02 | Re-enabling Seller edit access after an event is closed — access is one-way (Active → ReadOnly) per event. |
| OOS03 | Seller role downgrade outside the context of an event close — this story is scoped to event-end triggers only. |

---

### Non-functional Requirements
| Requirement | Description |
|---|---|
| Performance | Role status update on event close must complete within 5 seconds regardless of the number of Sellers assigned to the event (expected scale: up to ~1,500 users). |
| Security | ReadOnly enforcement must be applied server-side. Client-side UI changes (hiding buttons) are supplementary only and must not be the sole access control mechanism. |
| Reliability | If the event end date trigger fails (e.g., scheduled job failure), the system must not leave Sellers in Active state indefinitely. A retry or fallback evaluation on next Admin login is acceptable. |

---

### Dev Handoff Notes

- Role model: the app DB (Prisma) holds a `UserEventRole` record per user per event with a `status` field (`ACTIVE` | `READ_ONLY`). The JWT does not contain role or status — resolved per-request from the DB.
- Downgrade trigger A (Admin close): when EP-02 US-02.04 transitions event status to `CLOSED`, run a batch update: `UPDATE UserEventRole SET status = 'READ_ONLY' WHERE eventId = <id> AND role = 'SELLER' AND status = 'ACTIVE'`.
- Downgrade trigger B (end date): implement as either a scheduled job (cron) evaluated daily, or lazily evaluated on each request when `event.endDate < now()`. Lazy evaluation is simpler for Phase 1 given scale.
- In-app notification (AC05): implement as a `Notification` record inserted to the DB per affected Seller. Low priority — can be deferred to a later sprint without blocking AC01–AC04.
- Server-side enforcement: all mutating API routes for products and orders must check `UserEventRole.status` and return `403 Forbidden` if `READ_ONLY`, regardless of UI state.
- Phase 2: role resolution logic is unchanged after Azure AD migration. The `UserEventRole` table is the authoritative source for access control.
