# US-01.02 Log In

### User Story ID
`US-01.02`

### User Story Name
Log In

### User Story
As a **user**, I want to log in with my email and passcode, so that I can browse, place orders and donate.

---

### Pre-conditions
| Pre-condition ID | Description |
|------------------|-------------|
| PR01 | The user has an existing account created via the Sign Up flow (US-01.01). |
| PR02 | The account is in an active state (not deactivated by Admin). |
| PR03 | The Log In page is publicly accessible without prior authentication. |

---

### Workflow/Activity Diagram

1. User navigates to the Log In page.
2. User enters email and 6-digit passcode, then submits.
3. System validates credentials against the auth provider (Supabase Auth).
4. **If credentials are invalid:** System displays a generic error. User remains on the Log In page.
5. **If the account is deactivated:** System blocks login and displays a contact message.
6. **If credentials are valid:** System issues a JWT (standard OIDC claims). Application resolves the user's role from the app DB for the active event.
7. **If an active event exists:** User is redirected to the appropriate view for their role (Buyer home, Seller dashboard, Admin dashboard).
8. **If no active event:** User is redirected to the previous event's donation results page.

---

### Suggested UI Specifications

| Field Name | Type | Mandatory | Default Value | Description |
|---|---|---|---|---|
| Email address | Text input | Yes | N/A | Standard email input. Max 254 characters. |
| Passcode | Numeric input (masked) | Yes | N/A | 6-digit numeric passcode. |
| Log In button | Primary button | Yes | N/A | Submits the form. Disabled while submission is in progress. |
| Sign Up link | Text link | No | N/A | Navigates to the Sign Up page (US-01.01). |

**Error and system messages:**

| Trigger | Message |
|---|---|
| Email not found or passcode incorrect | "Email or passcode is incorrect." |
| Account deactivated | "Your account has been deactivated. Please contact the event organiser." |

### Business Acceptance Criteria

**AC01** — Successful login — active event

**Given** a user with a valid, active account is on the Log In page,
**When** they submit correct credentials and an active event exists,
**Then** the system authenticates them and redirects them to the view appropriate to their role for that event (Buyer home, Seller dashboard, or Admin dashboard).

**AC02** — Successful login — no active event

**Given** a user with a valid, active account submits correct credentials,
**When** there is no active event,
**Then** the user is authenticated and redirected to the previous event's donation results page.

**AC03** — Invalid credentials

**Given** a user submits the Log In form,
**When** the email is not found or the passcode is incorrect,
**Then** the system displays "Email or passcode is incorrect." and does not reveal which field is wrong.

**AC04** — Deactivated account

**Given** a user's account has been deactivated by an Admin,
**When** the user attempts to log in,
**Then** the system blocks authentication and displays "Your account has been deactivated. Please contact the event organiser."

**AC05** — Role resolution from app DB

**Given** a user is authenticated,
**When** the system determines the user's view,
**Then** the role is resolved from the app DB (not from JWT claims) against the active event. The JWT contains only standard OIDC claims (`sub`, `email`, `iat`, `exp`, `aud`, `iss`).

**AC06** — Admin login

**Given** an Admin account exists (seeded in code or promoted by another Admin),
**When** the Admin submits correct credentials,
**Then** the Admin is authenticated and redirected to the Admin dashboard. Admin accounts cannot be created via the Sign Up page.

### Out of Scope
| OOS ID | Description |
|--------|-------------|
| OOS01 | Passcode reset / forgot passcode — out of scope for Phase 1. |
| OOS02 | Social or SSO login (e.g., Google, Microsoft) — Phase 2 will use Azure AD / Entra ID SSO; no code changes to the login flow are expected at that point. |
| OOS03 | Multi-factor authentication (MFA) — not required for Phase 1. |
| OOS04 | "Remember me" / persistent sessions — session duration follows auth provider defaults. |

---

### Non-functional Requirements
| Requirement | Description |
|---|---|
| Security | Credentials are transmitted over HTTPS only. Generic error messaging prevents enumeration of valid email addresses. |
| Performance | Login form submission should complete within 2 seconds under normal load. |
| Cross-platform | Form must be fully functional on mobile and desktop browsers (Chrome, Safari, Edge — latest two versions). |

---

### Dev Handoff Notes

- Auth provider: Supabase Auth (Phase 1). Use standard `signInWithPassword` method (passcode is treated as the password string).
- JWT claims: `sub`, `email`, `iat`, `exp`, `aud`, `iss` only. Role is never stored in the JWT.
- After authentication, the application must look up the user's role in the app DB (Prisma) scoped to the current active event to determine redirect destination and access level.
- Deactivated account check: store an `isActive` flag on the user record in the app DB and check it on every login before allowing access.
- Phase 2: Supabase Auth OIDC config swapped for Azure AD / Entra ID. Role resolution logic in the app DB is unchanged.
