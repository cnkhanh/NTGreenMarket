# EP-03 — Manage Users

## Goal

Allow Admins to assign event-scoped Seller and Buyer roles to registered users without account deactivation. Admin role is global and managed separately.

## Vision

A streamlined admin interface where Admins can:
- View all registered users with key attributes (Name, Email, Location)
- Assign Seller roles to users per event via bulk multi-select
- Revoke Seller roles (users revert to Buyer)
- View all users with search (name/email) and location filter
- Promote other users to Admin (global, unrestricted)
- Automatically revert Seller → Buyer when events end

## In Scope (Phase 1)

**User Management:**
- Assign Seller role to registered users for a specific event (bulk via multi-select)
- Revoke Seller role for a specific event (user reverts to Buyer)
- Change or reassign user roles per event
- View all users with search (name/email) and location filter
- Promote any user to Admin (unrestricted; system ensures ≥1 Admin always exists)
- Promote any user to remove Admin status (with last-Admin guard)

**Auto-Revert:**
- When an event ends (status = closed), all Seller roles for that event automatically revert to Buyer
- Users retain Buyer role and access to view old shops
- Users can be re-assigned as Seller in new events
- No manual deactivation; access control is entirely role-based

**UI & Data:**
- User list columns: Name, Email, Location
- Search by name or email (text search, partial match)
- Filter by location (Ha Noi / Da Nang / Ho Chi Minh)
- Per-event role assignment view (select event → multi-select users → assign role)

## Out of Scope

- Account deactivation (removed entirely — Phase 1 uses role-only model)
- Bulk import via CSV (Phase 2)
- Audit logging / change history (Phase 2)
- RBAC expansion or department-level permissions (Phase 2)
- Email notifications to users on role changes (Phase 2, low priority)
- Role-based approval workflows (Phase 2)

## Roles Involved

- **Admin** — global role; can perform all user management operations; access all events
- **Seller** — event-scoped, assigned by Admin; can create/edit shops for that event only
- **Buyer** — default role; all users have Buyer access (Admin and Seller users also have Buyer access)

## Key Business Rules

### Admin Promotion & Demotion
- **No restrictions:** Any Admin can promote any user to Admin
- **Self-demotion allowed:** Admin can demote self or other Admins
- **Minimum Admin requirement:** System must always have ≥1 Admin (enforced on demotion)
- **Demotion UI hint:** Show warning if demoting the last Admin

### Role Model
- **Flat roles:** Users have one primary role per event (Seller or Buyer); Admin is global and independent
- **Event-scoped assignment:** User can be Seller in Event A and Buyer in Event B simultaneously
- **All users are Buyers:** Seller role is additive; Sellers retain Buyer access to browse and order
- **Per-request evaluation:** Role is resolved from app DB on every authenticated request; access changes take effect immediately

### Automatic Role Reversion
- **Event closure trigger:** When event status = closed, all Seller roles for that event automatically revert to Buyer
- **Effect:** User loses shop edit access but retains browse/view access to old shops
- **User notification:** Not Phase 1 (Phase 2 email notifications)
- **Re-assignment:** User can be assigned as Seller in future events

### Manual Role Revocation
- **Admin action:** Admin can manually revoke Seller → Buyer anytime (independent of event status)
- **Effect:** Immediate; user's shop becomes read-only on next request
- **Reversible:** User can be re-assigned as Seller later

## Dependencies

- **EP-01 (User Authentication & Access):** Users must self-register before Admins can assign roles. Role evaluation logic (per-request, from app DB, standard JWT) must align.
- **EP-02 (Manage Event):** Event status changes (close) trigger auto-revert. Coordination needed on event closure flow.
- **EP-05 (Shop & Product Management):** Seller role guards shop creation/editing. Post-event read-only behavior must align.
- **Prisma ORM & Supabase:** Data persistence for users, roles, and events.

## Design & Implementation Notes

### Data Model Overview

**Core tables:**
- `User` — email, name, location, is_admin (global)
- `Event` — name, status (DRAFT | PUBLISHED | CLOSED), start_date, end_date
- `UserRole` — user_id, event_id, role (SELLER | BUYER), assigned_at, assigned_by

**Constraints:**
- Email unique on User
- At least one Admin must exist globally (enforced on demotion)
- One role per user per event (UserRole unique on user_id, event_id)
- Event dates must be valid (start < end)

### API Endpoints (Phase 1)

| Method | Endpoint | Purpose | Auth |
|---|---|---|---|
| GET | `/api/admin/users` | List users with search/filter | Admin only |
| GET | `/api/admin/events/{eventId}/users` | List users for a specific event with roles | Admin only |
| PATCH | `/api/admin/events/{eventId}/users/{userId}/role` | Assign/change role for user in event | Admin only |
| DELETE | `/api/admin/events/{eventId}/users/{userId}/role` | Revoke role for user in event (revert to Buyer) | Admin only |
| POST | `/api/admin/users/{userId}/promote` | Promote user to Admin (global) | Admin only |
| DELETE | `/api/admin/users/{userId}/promote` | Demote user from Admin (global, with guard) | Admin only |

### Admin Dashboard UI Flow

**Entry:** Admin logs in → Dashboard (EP-02) → Navigate to "Manage Users"

**Flows:**
1. **View Users:** List all users with name, email, location; search by name/email; filter by location
2. **Assign Seller Role (Bulk):**
   - Select event from dropdown
   - Display table of users (Sellers shown with visual cue)
   - Multi-select users
   - Click "Assign Seller Role" → confirm → batch update
3. **Revoke Role:**
   - Select event
   - View user with current role
   - Click "Revoke Seller Role" → confirm → revert to Buyer
4. **Promote to Admin:**
   - User row context menu or detail page
   - Click "Promote to Admin" → confirm (warn if demoting last Admin) → update
