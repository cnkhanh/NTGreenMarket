# US-03.01 — Assign Seller Role to Users (Bulk)

### User Story ID
`US-03.01`

### User Story Name
Assign Seller Role to Users (Bulk, Per-Event)

### User Story
As an **Admin**, I want to assign the Seller role to multiple registered users for a specific event via bulk multi-select, so that I can onboard sellers efficiently without assigning roles one at a time.

---

### Pre-conditions
| Pre-condition ID | Description |
|------------------|-------------|
| PR01 | The Admin is authenticated and has Admin role (User.is_admin = true). |
| PR02 | At least one event exists in the system (status = PUBLISHED or CLOSED). |
| PR03 | At least one registered user exists (self-registered via EP-01). |
| PR04 | The selected users are not already assigned the Seller role for the selected event. |

---

### Workflow/Activity Diagram

1. Admin navigates to the Admin Dashboard → "Manage Users" section.
2. System displays a view with:
   - Event selector dropdown (lists all events ordered by date, default: current active event)
   - User list table (columns: Name, Email, Location, Current Role for selected event)
3. Admin selects an event from the dropdown.
4. System refreshes the user list table to show all registered users and their current role for the selected event.
5. Admin reviews the user list and identifies users to assign as Sellers.
6. Admin selects multiple users via checkboxes in the table (multi-select, no limit).
7. Admin clicks "Assign Seller Role" button (appears only when ≥1 user is selected).
8. System displays a confirmation dialog:
   - Summary: "You are about to assign Seller role to [N] users for event [Event Name]. This will grant them access to create and manage a shop. Continue?"
   - Users listed in the dialog (scrollable if many)
   - Buttons: "Cancel" and "Confirm"
9. **If Admin clicks Cancel:** Dialog closes, user selections remain.
10. **If Admin clicks Confirm:**
    - System creates or updates `UserRole` records for each selected user: `user_id`, `event_id = [selected event]`, `role = SELLER`, `assigned_at = NOW()`, `assigned_by = [current Admin user_id]`
    - System validates: each user exists, no duplicate role assignments per user per event
    - If all validations pass, system updates all records atomically
    - System displays success message: "Seller role assigned to [N] users for [Event Name]."
    - User selections are cleared; table refreshes to show updated roles
11. **If validation fails for any user:**
    - System displays error message: "Failed to assign role to [N] user(s). Reason: [details]. Remaining users were not updated."
    - System rolls back all changes (atomic transaction)
    - User selections remain; Admin can retry or adjust selection

### Suggested UI Specifications

**Event Selector:**
| Field Name | Type | Mandatory | Default Value | Description |
|---|---|---|---|---|
| Event selector | Dropdown | Yes | Current active event (or first in list) | Lists all events by name and date. Format: "Event Name (dd MMM - dd MMM yyyy)". |

**User List Table:**
| Column | Type | Sortable | Filterable | Description |
|---|---|---|---|---|
| Checkbox (multi-select) | Checkbox | No | No | Selects user for bulk assignment. |
| Name | Text | Yes (A-Z) | Yes (search) | User's full name. |
| Email | Text | Yes (A-Z) | Yes (search) | User's email address. |
| Location | Text | Yes (A-Z) | Yes (filter dropdown) | Ha Noi, Da Nang, or Ho Chi Minh. |
| Current Role (for event) | Badge | Yes | Yes (filter: Buyer / Seller) | Current role for selected event. Already-assigned Sellers shown with checkmark or "already assigned" label. |

**Table Controls:**
- Search box (name/email, real-time filter, partial match)
- Location filter dropdown (multi-select: Ha Noi, Da Nang, Ho Chi Minh; "All locations")
- "Clear filters" link
- Pagination or infinite scroll (up to 100 users per page; Phase 2 consideration: virtual scroll for large lists)
- "Select all on this page" / "Deselect all" shortcuts

**Confirmation Dialog:**
| Element | Description |
|---|---|
| Title | "Assign Seller Role" |
| Summary text | "You are about to assign Seller role to [N] users for event [Event Name]. This will grant them access to create and manage a shop. Continue?" |
| User list | Scrollable list of selected users (columns: Name, Email, Location). |
| Cancel button | Secondary button, closes dialog, retains selections. |
| Confirm button | Primary button, submits bulk assignment. Disabled during submission. |

**Success Message:**
- Toast or inline message: "Seller role assigned to [N] users for [Event Name]."
- Auto-dismiss after 3 seconds or manual close.

**Error Message:**
- Alert or toast: "Failed to assign role to [N] user(s). Reason: [details]. Remaining users were not updated."
- Details include: list of failed user_ids and specific error (e.g., "User does not exist").
- Inline validation: if user does not exist, show error badge in Name column; disable checkbox or show warning tooltip.

### Business Acceptance Criteria

**AC01 — Successful bulk assignment**

**Given** an Admin is on the Manage Users page with an event selected,
**When** the Admin selects multiple users (checkboxes) and clicks "Assign Seller Role",
**Then** the system displays a confirmation dialog listing the selected users and the target event.

**AC02 — Confirmation flow**

**Given** the confirmation dialog is open,
**When** the Admin clicks "Confirm",
**Then** the system assigns the Seller role to all selected users for the selected event, displays a success message, and refreshes the user list table.

**AC03 — Rollback on partial failure**

**Given** the system is attempting to assign Seller role to [N] users,
**When** at least one user fails validation (e.g., does not exist),
**Then** the system rolls back all changes (no user is assigned), displays an error message with failure details, and retains user selections for retry.

**AC04 — Prevent duplicate role assignment**

**Given** a user is already assigned the Seller role for the selected event,
**When** the Admin selects that user again and submits,
**Then** the system skips the duplicate (idempotent) and displays success (no change made). Confirm with PO if idempotency is preferred.

**AC05 — Event validation**

**Given** the Admin selects an event,
**When** the event no longer exists or status is neither PUBLISHED nor CLOSED,
**Then** the system displays an error: "Event is no longer available. Please select another event."

**AC06 — Role assignment audit trail (logged, not displayed Phase 1)**

**Given** the system successfully assigns roles,
**When** the operation completes,
**Then** the system logs: user_id, event_id, role, assigned_at, assigned_by (Admin's user_id).

### Non-functional Requirements
| Requirement | Description |
|---|---|
| Performance | Bulk assignment of up to 100 users should complete within 3 seconds under normal load. |
| Scalability | System must support assigning Seller role to up to 1,500 users per event. |
| Atomicity | All role assignments in a single bulk operation must succeed together or fail together (no partial updates). |
| Concurrency | If two Admins assign the same user simultaneously, last-write-wins is acceptable; display retry message to the slower Admin. |
| UI Responsiveness | Multi-select and table filtering should remain responsive (no blocking UI freezes). |
| Accessibility | Checkboxes, buttons, and dialogs must follow WCAG 2.1 AA standards. Multi-select accessible via keyboard (Space to select, Enter to submit). |

### Dev Handoff Notes

**API Endpoint:**
```
PATCH /api/admin/events/{eventId}/users/roles/bulk
Content-Type: application/json

Request Body:
{
  "user_ids": [
    "user-id-1",
    "user-id-2",
    "user-id-3"
  ],
  "role": "SELLER"
}

Response (200 OK):
{
  "success": true,
  "assigned_count": 3,
  "message": "Seller role assigned to 3 users for event [Event Name]."
}

Response (400 Bad Request - Partial Failure):
{
  "success": false,
  "assigned_count": 2,
  "failed_count": 1,
  "failures": [
    {
      "user_id": "user-id-3",
      "error": "User does not exist"
    }
  ],
  "message": "Failed to assign role to 1 user(s). Changes rolled back."
}
```

**Database Logic:**
- Query `User` table: verify all user_ids exist
- Query `Event` table: verify event_id exists and status IN (PUBLISHED, CLOSED)
- Query `UserRole` table: check for duplicates (user_id, event_id, role)
- Insert/Update `UserRole` records in a single transaction; rollback on any error
- Use upsert for idempotent behavior (if user already has role, update timestamp only)
- Log operation to server logs (timestamp, admin_id, user_ids, event_id, role)

**Client-Side Validation:**
- Disable "Assign Seller Role" button if no users are selected
- Show inline warning next to users that fail validation (error badge)
- Real-time search/filter on client to reduce API calls during table browsing
- Disable Confirm button during submission; show spinner

**Edge Cases:**
- Admin selects a user, then that user is deleted before submission → API returns error, rollback, show message
- Two Admins assign the same user simultaneously → last write wins; slower Admin sees "already assigned" (idempotent)
- Admin clicks Assign, then navigates away before confirmation → pending request cancels (AbortController)
- Event is deleted between dropdown selection and confirmation → API returns "Event not found"
- User already has Seller role → upsert updates timestamp, return idempotent success

**Testing Strategy:**
- Unit: Role validation, constraint checks, transaction rollback
- Integration: Multi-user assignment, event validation, idempotent behavior
- E2E: Full flow from user selection to success confirmation and table refresh
- Load: 100+ user bulk assignment under simulated load
- Concurrency: Two Admins assigning same user simultaneously
