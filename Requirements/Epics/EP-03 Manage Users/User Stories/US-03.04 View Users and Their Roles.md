# US-03.04 — View Users and Their Roles

### User Story ID
`US-03.04`

### User Story Name
View Users and Their Roles (List, Search, Filter)

### User Story
As an **Admin**, I want to view all registered users with their attributes and current roles for a specific event, with the ability to search by name/email and filter by location, so that I have full visibility of participants and can quickly identify users for role assignment or management.

---

### Pre-conditions
| Pre-condition ID | Description |
|------------------|-------------|
| PR01 | The Admin is authenticated and has Admin role (User.is_admin = true). |
| PR02 | At least one registered user exists. |
| PR03 | At least one event exists in the system. |

---

### Workflow/Activity Diagram

1. Admin navigates to Admin Dashboard → "Manage Users" section.
2. System loads the user management page and displays:
   - Event selector dropdown (default: current active event or first available)
   - User list table with columns: Name, Email, Location, Current Role (for selected event)
   - Search box (placeholder: "Search by name or email")
   - Location filter dropdown (multi-select: All, Ha Noi, Da Nang, Ho Chi Minh)
   - Results summary: "Showing X of Y users" or "X users match your filters"
   - Pagination controls (Phase 1: pagination; Phase 2: virtual scroll for large lists)
3. Admin selects an event from the dropdown (if not default).
4. System queries users and their roles for the selected event:
   - Fetch `User` records with `location`
   - Fetch `UserRole` records for selected `event_id`
   - Join results to show each user's role for the event
5. System displays all users in the table (up to page limit, e.g., 25 per page).
6. Admin can search by typing in the search box (real-time filtering, partial match on name or email).
7. System displays filtered results immediately (client-side or server-side, clarify performance requirements).
8. Admin can filter by location by selecting from the dropdown.
9. System displays users matching both search and location filters.
10. Admin can sort by any column (click column header to sort A-Z or Z-A).
11. Admin can select users via checkboxes for bulk actions (US-03.01).

### Business Acceptance Criteria

**AC01 — View all users for an event**

**Given** an Admin selects an event from the dropdown,
**When** the user list table loads,
**Then** all registered users are displayed with their attributes (Name, Email, Location) and current role for the selected event.

**AC02 — Real-time search by name**

**Given** an Admin types "john" in the search box,
**When** results update in real-time,
**Then** only users with "john" in their name are displayed (case-insensitive, partial match).

**AC03 — Real-time search by email**

**Given** an Admin types "example@company.com" in the search box,
**When** results update in real-time,
**Then** the user with that email is displayed.

**AC04 — Filter by location**

**Given** an Admin selects "Ha Noi" from the location filter,
**When** the table updates,
**Then** only users with location = Ha Noi are displayed.

**AC05 — Combine search and filters**

**Given** an Admin searches for "john" AND filters by location "Ha Noi",
**When** the table updates,
**Then** only users with "john" in their name AND location = Ha Noi are displayed.

**AC06 — Clear filters**

**Given** an Admin has applied search and location filters,
**When** the Admin clicks "Clear filters",
**Then** all filters are cleared; the table displays all users for the selected event.

**AC07 — Sort by column**

**Given** an Admin clicks the "Name" column header,
**When** the table re-renders,
**Then** users are sorted A-Z by name. Clicking again sorts Z-A. A visual indicator (▲/▼) appears next to the column name.

**AC08 — Multi-select for bulk actions**

**Given** an Admin selects multiple users via checkboxes,
**When** the Admin clicks "Assign Seller Role",
**Then** the bulk assignment flow is triggered (see US-03.01).

**AC09 — Display current role per event**

**Given** an Admin selects Event A,
**When** the table loads,
**Then** the "Current Role" column shows each user's role for Event A. If the Admin selects Event B, roles update to show roles for Event B.

**AC10 — Admin status is visible**

**Given** a user has global Admin status (User.is_admin = true),
**When** the Admin views the user list,
**Then** the user is displayed with role badge = "Admin" (gold/orange). Admin status is independent of per-event roles.

**AC11 — Pagination**

**Given** more than 25 users exist (configurable),
**When** the table loads,
**Then** users are paginated; pagination controls display "Page 1 of X". Admin can navigate pages.

### Non-functional Requirements
| Requirement | Description |
|---|---|
| Performance | List should load within 2 seconds for up to 1,500 users. Search/filter debounce at 300ms. Pagination load within 1 second. |
| Responsiveness | Table must be responsive on mobile (sticky header, horizontal scroll for columns if needed). |
| Accessibility | Table must follow WCAG 2.1 AA standards. Keyboard navigation: Tab to focus columns/controls, arrow keys to navigate rows, Space/Enter to select checkboxes, sort via keyboard. Screen reader support for table structure and sort order. |
| Data Freshness | User list reflects changes within 5 seconds (client refresh or auto-refresh). Clarify with PO. |
| Concurrency | If another Admin makes changes while current Admin is viewing, display a "Refresh data" prompt or auto-refresh (clarify). |
| Pagination | Efficient pagination (server-side cursor or offset-based); avoid loading all users at once. |
| Search | Search results returned within 500ms for 1,500 users (optimize with indexes). |

### Dev Handoff Notes

**API Endpoint:**

```
GET /api/admin/users?event_id=[eventId]&search=[searchTerm]&location=[location]&page=[pageNum]&limit=[pageSize]&sort_by=[column]&sort_order=[asc|desc]

Headers: Authorization: Bearer [JWT token]

Query Parameters:
- event_id (required): Event ID for role context
- search (optional): Search term for name/email (partial, case-insensitive)
- location (optional): Ha Noi | Da Nang | Ho Chi Minh (can be comma-separated for multi-select)
- role (optional): ADMIN | SELLER | BUYER (can be comma-separated for multi-select)
- page (optional): Page number, default = 1
- limit (optional): Results per page, default = 25, max = 100
- sort_by (optional): name | email | location | role, default = name
- sort_order (optional): asc | desc, default = asc
```

**Database Queries:**

**Base Query (user list for event with roles):**
```sql
SELECT
  u.id,
  u.name,
  u.email,
  u.location,
  u.is_admin,
  ur.role AS current_role,
  ur.assigned_at AS role_assigned_at,
  ur.assigned_by AS role_assigned_by
FROM users u
LEFT JOIN user_roles ur ON u.id = ur.user_id AND ur.event_id = ?
WHERE 1=1
  AND (? IS NULL OR LOWER(u.name) LIKE ? OR LOWER(u.email) LIKE ?)  -- search
  AND (? IS NULL OR u.location = ?)  -- location filter
  AND (? IS NULL OR ur.role = ?)  -- role filter (if filtering by SELLER/BUYER)
ORDER BY u.name ASC
LIMIT ? OFFSET ?
```

**Indexes (for performance):**
- Index on User.email (for search)
- Index on User.location
- Index on UserRole(user_id, event_id)
- Full-text search index on User.name + User.email (if using PostgreSQL)

**Client-Side Logic:**
- Debounce search input at 300ms before sending API request
- Cache event list (small dataset)
- Implement pagination: fetch page on demand
- Implement sorting: re-sort results on column click (or re-fetch if server-side)

**Edge Cases:**
- Event has no users → display "No users registered yet."
- Search returns no results → display "No users match your search and filters."
- User is global Admin (is_admin = true) → show "Admin" in current_role column (independent of UserRole)
- Concurrent update: Another Admin assigns role while current Admin views list → depends on refresh strategy (auto vs. manual)

**Testing Strategy:**
- Unit: Search logic, filter combination, pagination math
- Integration: User + UserRole join, role resolution per event
- E2E: Full view flow with search, filter, sort, pagination
- Performance: Load 1,500 users, search response time, pagination load time
- Accessibility: Keyboard navigation, screen reader support
