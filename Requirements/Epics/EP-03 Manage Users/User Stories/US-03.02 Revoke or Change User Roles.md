# US-03.02 — Revoke or Change User Roles (Per-Event)

### User Story ID
`US-03.02`

### User Story Name
Revoke or Change User Roles (Per-Event)

### User Story
As an **Admin**, I want to revoke or change a user's Seller role for a specific event, so that I can control seller access per event and adapt to changing business needs.

---

### Pre-conditions
| Pre-condition ID | Description |
|------------------|-------------|
| PR01 | The Admin is authenticated and has Admin role (User.is_admin = true). |
| PR02 | At least one event exists in the system. |
| PR03 | At least one user is assigned a Seller role for the selected event. |

---

### Workflow/Activity Diagram

**Scenario 1: Revoke Seller Role (Single User, Single Event)**

1. Admin navigates to Admin Dashboard → "Manage Users".
2. Admin selects an event from the dropdown.
3. System displays user list table with columns: Name, Email, Location, Current Role (for selected event).
4. Admin identifies a user with Seller role and clicks "Revoke" (context menu or inline button in user row).
5. System displays a confirmation dialog:
   - Title: "Revoke Seller Role"
   - Message: "You are about to revoke Seller role from [User Name] for event [Event Name]. User will revert to Buyer. Continue?"
   - Buttons: "Cancel" and "Confirm"
6. **If Admin clicks Cancel:** Dialog closes; no changes.
7. **If Admin clicks Confirm:**
   - System deletes the `UserRole` record for user_id, event_id
   - User's role for this event reverts to BUYER (default)
   - User's shop (if exists) transitions to read-only on next request
   - System displays success message: "Seller role revoked from [User Name] for [Event Name]."
   - Table refreshes to show updated role (reverts to Buyer)

**Scenario 2: Change Role (e.g., Seller → Buyer or vice versa)**

1. Admin navigates to user detail view or right-clicks user in table.
2. Admin selects "Change Role" from context menu.
3. System displays a dropdown or role selection panel:
   - Options: Buyer, Seller (radio buttons or dropdown)
   - Current role highlighted
4. Admin selects the new role.
5. System displays a confirmation dialog:
   - Title: "Change Role"
   - Message: "You are about to change [User Name]'s role from [Old Role] to [New Role] for event [Event Name]. Continue?"
   - Buttons: "Cancel" and "Confirm"
6. **If Admin clicks Cancel:** Dialog closes; no changes.
7. **If Admin clicks Confirm:**
   - System updates the `UserRole` record: role = [New Role], updated_at = NOW()
   - If updating, verify no conflicts (e.g., if changing from Seller, shop transitions to read-only)
   - Update the record and display success message
   - Table refreshes to show updated role

### Business Acceptance Criteria

**AC01 — Successful revocation**

**Given** a user has Seller role for the selected event,
**When** the Admin clicks "Revoke" and confirms,
**Then** the Seller role is removed for that event; the user's role reverts to Buyer. Shop access becomes read-only.

**AC02 — Successful role change**

**Given** a user has Seller role for the selected event,
**When** the Admin selects "Change Role" → "Buyer" and confirms,
**Then** the user's role is updated to Buyer for that event. Shop access is revoked.

**AC03 — Confirmation required**

**Given** the Admin clicks "Revoke" or "Change Role",
**When** a confirmation dialog appears,
**Then** the operation does not proceed until the Admin clicks "Confirm". Clicking "Cancel" discards changes.

**AC04 — Shop state transitions on role revocation**

**Given** a Seller has an active shop for the selected event,
**When** the Seller's role is revoked for that event,
**Then** the shop is not deleted; it transitions to read-only. The user can view the shop but cannot edit products or change shop status.

**AC05 — Post-event read-only is independent from revocation**

**Given** an event has ended (status = CLOSED),
**When** a user's Seller role is revoked,
**Then** the user's shop becomes read-only (same outcome as event closure auto-revert). Both manual revocation and auto-revert result in read-only access.

---

### Non-functional Requirements
| Requirement | Description |
|---|---|
| Performance | Role revocation or change should complete within 1 second under normal load. |
| Atomicity | Role change is atomic; if the operation fails, no change is persisted. |
| Concurrency | If two Admins revoke the same user's role simultaneously, the second operation should detect the conflict (role already revoked) and return idempotent success. |
| Immediate Effect | Role changes take effect immediately on next API request (no session caching). If a Seller is logged in and their role is revoked, their next API call should deny shop edit access. |
| UI Responsiveness | Dialog should appear within 500ms. Confirmation should not block other UI interactions. |
| Accessibility | Dialog and role selector must follow WCAG 2.1 AA standards. Keyboard navigation: Tab between elements, Enter to confirm, Esc to cancel. |

### Dev Handoff Notes

**API Endpoints:**

**Revoke Seller Role:**
```
DELETE /api/admin/events/{eventId}/users/{userId}/role

Headers: Authorization: Bearer [JWT token]

Response (200 OK):
{
  "success": true,
  "user_id": "user-id",
  "event_id": "event-id",
  "revoked_role": "SELLER",
  "new_role": "BUYER",
  "message": "Seller role revoked from [User Name] for [Event Name]. User reverted to Buyer."
}

Response (400 Bad Request):
{
  "success": false,
  "error": "User does not have Seller role for this event (already Buyer)"
}

Response (403 Forbidden):
{
  "success": false,
  "error": "Not authorized to revoke roles"
}

Response (404 Not Found):
{
  "success": false,
  "error": "User or event not found"
}
```

**Change Role:**
```
PATCH /api/admin/events/{eventId}/users/{userId}/role

Content-Type: application/json
Headers: Authorization: Bearer [JWT token]

Request Body:
{
  "role": "SELLER" | "BUYER"
}

Response (200 OK):
{
  "success": true,
  "user_id": "user-id",
  "event_id": "event-id",
  "new_role": "SELLER",
  "message": "[User Name]'s role changed to Seller for [Event Name]."
}

Response (400 Bad Request):
{
  "success": false,
  "error": "User does not exist"
}

Response (409 Conflict):
{
  "success": false,
  "error": "Role has already been changed by another Admin. Please refresh and try again."
}
```

**Database Logic:**

**Revoke Role:**
- Query `UserRole` table: WHERE user_id = ? AND event_id = ?
- If not found, return 400 error: "User does not have Seller role for this event"
- Delete the record (hard delete acceptable for MVP)
- Update `Shop` table state to read-only if needed (dependent on EP-05 implementation)
- Log operation: timestamp, admin_id, user_id, event_id, old_role, action = "revoked"

**Change Role:**
- Query `UserRole` table: WHERE user_id = ? AND event_id = ?
- If not found, create new UserRole (first assignment)
- If exists, update role field: role = [new_role], updated_at = NOW()
- Log operation: timestamp, admin_id, user_id, event_id, old_role, new_role, action = "changed"

**Shop State Transition (EP-05 Integration):**
- On role revocation (SELLER → BUYER), shop becomes read-only (enforced via EP-05 role check)
- On role change (e.g., SELLER → BUYER), shop becomes read-only
- Clarify shop state model with EP-05 owner

**Edge Cases:**
- Admin revokes own role while logged in → next API request denies access; show "Access revoked" message
- Event is deleted between role change dialog and submission → return 404
- Two Admins revoke same user's role simultaneously → second admin sees "already revoked" or idempotent success
- User already has desired role → return idempotent success (no change)

**Testing Strategy:**
- Unit: Role validation, state transitions
- Integration: Revocation with shop state change, role change with concurrent edits
- E2E: Full revocation and role change flows with immediate effect verification
- Concurrency: Simultaneous role changes to same user
- Session: Verify logged-in user loses access immediately after role revocation
