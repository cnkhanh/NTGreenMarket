# US-03.05 — Promote User to Admin & Manage Global Admin Role

### User Story ID
`US-03.05`

### User Story Name
Promote User to Admin (Unrestricted, Global Role) & Demote from Admin

### User Story
As an **Admin**, I want to promote any registered user to Admin (unrestricted, no approval workflow) and demote Admins (with safeguards), so that I can delegate platform management responsibilities without barriers and maintain operational control.

---

### Pre-conditions
| Pre-condition ID | Description |
|------------------|-------------|
| PR01 | The Admin is authenticated and has Admin role (User.is_admin = true). |
| PR02 | At least one registered user exists to promote. |
| PR03 | (For demotion) At least two active Admins exist (to prevent demoting last Admin). |

---

### Workflow/Activity Diagram

**Scenario 1: Promote User to Admin (via User List)**

1. Admin navigates to Admin Dashboard → "Manage Users".
2. Admin selects an event from the dropdown.
3. System displays user list with roles for selected event.
4. Admin identifies a user to promote (e.g., a Seller or Buyer) and clicks "Promote to Admin" (context menu or inline button).
5. System displays a confirmation dialog:
   - Title: "Promote to Admin"
   - Message: "You are about to promote [User Name] to Admin. They will have full platform management access across all events. Continue?"
   - User info: Name, email, current role
   - Buttons: "Cancel" and "Confirm"
6. **If Admin clicks Cancel:** Dialog closes; no changes.
7. **If Admin clicks Confirm:**
   - System updates `User.is_admin = true`, `promoted_at = NOW()`, `promoted_by = [current Admin user_id]`
   - System displays success message: "[User Name] has been promoted to Admin."
   - User list refreshes; user's Admin status is visible in all event rows

**Scenario 2: Demote Admin (via Change Role or Context Menu)**

1. Admin clicks "Demote from Admin" on an Admin user (or accesses demotion via change role flow).
2. System displays a confirmation dialog:
   - Title: "Demote from Admin"
   - Message: "You are about to demote [User Name] from Admin to regular user. They will lose platform management access. Continue?"
   - If user is the last active Admin: "Warning: [User Name] is the last active Admin. Demotion will leave the system without an Admin. Promote another user to Admin first."
   - Buttons: "Cancel" and "Confirm" (Confirm disabled if last Admin)
3. **If Admin clicks Cancel:** Dialog closes; no changes.
4. **If Admin clicks Confirm:**
   - System updates `User.is_admin = false`
   - System displays success message: "[User Name] has been demoted from Admin."
   - User list refreshes; user's role shows as Seller or Buyer (per-event role)

**Scenario 3: Self-Promotion or Self-Demotion (Edge Case)**

1. Admin attempts to promote/demote themselves.
2. System evaluates the action:
   - **Promotion:** If already Admin, return idempotent success
   - **Demotion:** If last Admin, block with error; otherwise allow
3. Same confirmation flow as above

### Business Acceptance Criteria

**AC01 — Successful promotion**

**Given** an Admin clicks "Promote to Admin" on a non-Admin user,
**When** the Admin confirms the action,
**Then** the user is promoted to Admin; User.is_admin = true; the user has full platform management access across all events; a success message is displayed.

**AC02 — No approval workflow required**

**Given** an Admin initiates a promotion,
**When** the Admin confirms,
**Then** the promotion is applied immediately. No approval, review, or waiting period is required.

**AC03 — Unrestricted promotion**

**Given** any Admin exists (no seniority or restrictions),
**When** that Admin promotes any user to Admin,
**Then** the promotion succeeds. No restrictions on who can promote or be promoted.

**AC04 — Confirmation required**

**Given** an Admin clicks "Promote to Admin" or "Demote from Admin",
**When** a confirmation dialog appears,
**Then** the operation does not proceed until the Admin clicks "Confirm". Clicking "Cancel" discards changes.

**AC05 — System maintains at least one Admin**

**Given** only one active Admin exists,
**When** that Admin attempts to be demoted,
**Then** the system displays an error and blocks demotion. "Cannot demote [User Name]. [User Name] is the last active Admin."

**AC06 — Admin role is global**

**Given** a user is promoted to Admin,
**When** the Admin views other events,
**Then** the user has Admin status for all events (Admin role is global, not per-event). User is displayed with Admin badge in all event contexts.

**AC07 — Idempotent promotion**

**Given** a user is already an Admin,
**When** another Admin promotes that user again,
**Then** the operation succeeds silently (idempotent) with success message: "[User Name] is already an Admin." (or simply returns 200).

**AC08 — Promoted user gains immediate access**

**Given** a user is promoted to Admin,
**When** that user logs in or makes an API request,
**Then** their next request evaluates User.is_admin from the app DB and grants Admin access. No session caching or re-login required.

**AC09 — Demoted user loses admin access immediately**

**Given** a user is demoted from Admin,
**When** that user makes an API request,
**Then** their next request evaluates User.is_admin = false and denies Admin access. User retains event-scoped role (Seller/Buyer) if assigned.

**AC10 — Audit trail for promotion/demotion (logged, not displayed Phase 1)**

**Given** an Admin promotes or demotes a user,
**When** the operation completes,
**Then** the system logs: user_id, promoted_by/demoted_by (Admin's user_id), is_admin, timestamp, action = "promoted" or "demoted".

| OOS05 | Bulk promotion of multiple users (Phase 1; one at a time). |
| OOS06 | Scheduled or delayed promotion/demotion (Phase 2). |

---

### Non-functional Requirements
| Requirement | Description |
|---|---|
| Performance | Promotion/demotion should complete within 1 second under normal load. |
| Atomicity | Promotion/demotion is atomic; if the operation fails, no change is persisted. |
| Immediate Effect | Admin access changes take effect on next API request (per-request role evaluation). |
| Concurrency | If two Admins promote the same user simultaneously, last-write-wins is acceptable (idempotent). |
| Accessibility | Dialog must follow WCAG 2.1 AA standards. Keyboard navigation: Tab, Enter to confirm, Esc to cancel. |
| Security | Only authenticated Admin users can access promotion/demotion endpoints. Standard JWT validation. |

### Dev Handoff Notes

**API Endpoints:**

**Promote User to Admin:**
```
POST /api/admin/users/{userId}/promote

Headers: Authorization: Bearer [JWT token]

Request Body:
{} (empty)

Response (200 OK):
{
  "success": true,
  "user_id": "user-id",
  "name": "John Doe",
  "is_admin": true,
  "promoted_at": "2024-03-28T10:00:00Z",
  "message": "[User Name] has been promoted to Admin."
}

Response (200 OK - Idempotent):
{
  "success": true,
  "user_id": "user-id",
  "name": "John Doe",
  "is_admin": true,
  "message": "[User Name] is already an Admin."
}

Response (400 Bad Request):
{
  "success": false,
  "error": "User does not exist"
}

Response (403 Forbidden):
{
  "success": false,
  "error": "Not authorized to promote users"
}

Response (404 Not Found):
{
  "success": false,
  "error": "User not found"
}
```

**Demote User from Admin:**
```
DELETE /api/admin/users/{userId}/promote

Headers: Authorization: Bearer [JWT token]

Response (200 OK):
{
  "success": true,
  "user_id": "user-id",
  "name": "John Doe",
  "is_admin": false,
  "message": "[User Name] has been demoted from Admin."
}

Response (200 OK - Idempotent):
{
  "success": true,
  "user_id": "user-id",
  "name": "John Doe",
  "is_admin": false,
  "message": "[User Name] is not an Admin."
}

Response (400 Bad Request):
{
  "success": false,
  "error": "Cannot demote the last active Admin. Promote another user to Admin first."
}

Response (403 Forbidden):
{
  "success": false,
  "error": "Not authorized to demote users"
}

Response (404 Not Found):
{
  "success": false,
  "error": "User not found"
}
```

**Database Logic (Global Admin):**

**Promote User:**
```typescript
- Query `User` WHERE id = ?
- If user not found, return 404
- If user.is_admin = true, return 200 (idempotent): user is already Admin
- Update: User.is_admin = true, promoted_at = NOW(), promoted_by = [admin_id]
- Log operation: timestamp, admin_id, user_id, action = "promoted"
```

**Demote Admin:**
```typescript
- Query `User` WHERE id = ?
- If user not found, return 404
- If user.is_admin = false, return 200 (idempotent): user is not Admin
- Count active Admins: SELECT COUNT(*) FROM User WHERE is_admin = true
- If count <= 1, return 400 error: "Cannot demote last Admin"
- Update: User.is_admin = false
- Log operation: timestamp, admin_id, user_id, action = "demoted"
```

**Client-Side Logic:**
- Show "Promote to Admin" button only if user.is_admin = false
- Show "Demote from Admin" button only if user.is_admin = true AND not last Admin
- Disable Confirm button during submission
- Display success message and refresh user row in table
- Show Admin badge in table when user.is_admin = true

**Edge Cases:**
- Promote already-Admin → return 200 (idempotent)
- Demote already-non-Admin → return 200 (idempotent)
- Demote last Admin → return 400 (block)
- Promote self → idempotent success (already Admin)
- Demote self as last Admin → return 400 (block)
- Two Admins promote same user simultaneously → last-write-wins (idempotent succeeds)
- Promoted user logged in elsewhere → gains Admin access on next API request (per-request role evaluation)

**Testing Strategy:**
- Unit: Admin flag check, promotion validation, last-Admin count
- Integration: Global Admin role, immediate effect on API request
- E2E: Full promotion and demotion flows from dialog to table update
- Concurrency: Simultaneous promotions/demotions of same user
- Session: Verify promoted user gains access immediately without re-login
- Edge case: Verify last Admin guard blocks demotion
